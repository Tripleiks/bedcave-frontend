# Docker Compose for Beginners: Your First Multi-Container Setup

If you've started dabbling with Docker, you've probably noticed that running containers manually with long `docker run` commands gets old — fast. Enter **Docker Compose**: a tool that lets you define and manage multi-container applications using a simple YAML file. One command to bring everything up, one command to tear it all down. It's a game changer, especially for homelab builders.

In this guide, we'll walk through the basics of Docker Compose, break down the anatomy of a `compose.yaml` file, and build a real working example from scratch.

---

## What Is Docker Compose?

Docker Compose is a tool for defining and running multi-container Docker applications. Instead of juggling multiple `docker run` commands with dozens of flags, you describe your entire stack in a single file — networks, volumes, environment variables, port mappings — and Compose handles the rest.

> **Note:** Older tutorials reference `docker-compose` (the standalone binary). Modern Docker installations include Compose as a plugin, so the command is now `docker compose` (no hyphen). We'll use the modern syntax throughout this guide.

---

## Installing Docker Compose

If you have **Docker Desktop** installed (Windows or macOS), you're already good to go — Compose comes bundled.

On **Linux**, if you installed Docker Engine, install the Compose plugin:

```bash
sudo apt-get update
sudo apt-get install docker-compose-plugin
```

Verify it's working:

```bash
docker compose version
```

You should see something like `Docker Compose version v2.x.x`.

## The Anatomy of a `compose.yaml` File

The core of Docker Compose is the `compose.yaml` file (formerly `docker-compose.yml` — both names still work). Let's break down the key sections:

```yaml
services:
  app:
    image: nginx:latest
    ports:
      - "8080:80"
    volumes:
      - ./html:/usr/share/nginx/html
    environment:
      - MY_VAR=hello
    restart: unless-stopped

volumes:
  mydata:

networks:
  mynetwork:
```

Here's what each section means:

- **`services`** — Defines each container in your stack. Each key under `services` is the name of a container.
- **`image`** — The Docker image to use.
- **`ports`** — Maps host port to container port (`host:container`).
- **`volumes`** — Mounts files or directories from your host into the container.
- **`environment`** — Sets environment variables inside the container.
- **`restart`** — Defines the restart policy (`no`, `always`, `unless-stopped`, `on-failure`).
- **`volumes` (top-level)** — Declares named volumes managed by Docker.
- **`networks` (top-level)** — Declares custom networks.

## Your First Real Example: WordPress + MySQL

Let's build something practical — a WordPress site backed by a MySQL database. This is a classic two-container setup and a great first project.

Create a new directory and your Compose file:

```bash
mkdir wordpress-demo && cd wordpress-demo
nano compose.yaml
```

Paste this in:

```yaml
services:
  db:
    image: mysql:8.0
    container_name: wordpress_db
    restart: unless-stopped
    environment:
      MYSQL_ROOT_PASSWORD: rootpassword
      MYSQL_DATABASE: wordpress
      MYSQL_USER: wpuser
      MYSQL_PASSWORD: wppassword
    volumes:
      - db_data:/var/lib/mysql
    networks:
      - wp_network

  wordpress:
    image: wordpress:latest
    container_name: wordpress_app
    restart: unless-stopped
    ports:
      - "8080:80"
    environment:
      WORDPRESS_DB_HOST: db:3306
      WORDPRESS_DB_NAME: wordpress
      WORDPRESS_DB_USER: wpuser
      WORDPRESS_DB_PASSWORD: wppassword
    volumes:
      - wp_data:/var/www/html
    depends_on:
      - db
    networks:
      - wp_network

volumes:
  db_data:
  wp_data:

networks:
  wp_network:
```

Now bring it up:

```bash
docker compose up -d
```

The `-d` flag runs everything in **detached mode** (in the background). Visit `http://localhost:8080` in your browser and you'll see the WordPress setup wizard. That's two containers, a shared network, persistent volumes, all running from one file.

## Essential Docker Compose Commands

Here are the commands you'll use most often:

```bash
# Start all services in detached mode
docker compose up -d

# Stop and remove all containers (keeps volumes)
docker compose down

# Stop and remove containers AND volumes (destructive!)
docker compose down -v

# View running services
docker compose ps

# View logs from all services
docker compose logs

# Follow logs from a specific service
docker compose logs -f wordpress

# Restart a specific service
docker compose restart wordpress

# Pull latest images
docker compose pull

# Rebuild images (if using build:)
docker compose build
```

## Using `.env` Files for Secrets

Hardcoding passwords in your `compose.yaml` is fine for local testing, but bad practice in general. Docker Compose automatically loads a `.env` file in the same directory:

```bash
# .env
MYSQL_ROOT_PASSWORD=supersecretroot
MYSQL_PASSWORD=supersecretwp
WP_PORT=8080
```

Then reference the variables in your Compose file:

```yaml
services:
  db:
    image: mysql:8.0
    environment:
      MYSQL_ROOT_PASSWORD: ${MYSQL_ROOT_PASSWORD}
      MYSQL_PASSWORD: ${MYSQL_PASSWORD}

  wordpress:
    ports:
      - "${WP_PORT}:80"
```

Make sure to add `.env` to your `.gitignore` so you don't accidentally commit secrets to a repository.

## Building a Custom Image with Compose

Sometimes you want to build your own image instead of pulling one from Docker Hub. Compose handles this too with the `build` directive:

```yaml
services:
  myapp:
    build:
      context: .
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    restart: unless-stopped
```

Your project structure would look like:

```
myapp/
├── compose.yaml
├── Dockerfile
└── src/
    └── index.js
```

Running `docker compose up --build` will build the image from your Dockerfile before starting the container — perfect for development workflows.

## Healthchecks and Dependencies

The `depends_on` option ensures one service starts before another, but it doesn't wait for the service to be *ready*. For that, use healthchecks:

```yaml
services:
  db:
    image: mysql:8.0
    healthcheck:
      test: ["CMD", "mysqladmin", "ping", "-h", "localhost"]
      interval: 10s
      timeout: 5s
      retries: 5

  wordpress:
    image: wordpress:latest
    depends_on:
      db:
        condition: service_healthy
```

Now WordPress won't start until MySQL is confirmed healthy. This is essential for production-like setups in your homelab.

## Tips for Homelab Users

A few practical tips if you're running Compose on a home server:

- **Use `restart: unless-stopped`** on all services so your stack survives reboots without needing manual intervention.
- **Pin your image versions** (e.g., `mysql:8.0` instead of `mysql:latest`) to avoid surprise breaking changes after an update.
- **Keep your Compose files organized** — one directory per stack, with the `compose.yaml` and `.env` file together.
- **Use named volumes** for anything you care about persisting. Bind mounts (`./data:/data`) are fine too, but named volumes are managed by Docker and easier to back up.
- **Watchtower** can automatically update your containers when new images are released — pair it with pinned tags for controlled updates.

## Conclusion

Docker Compose takes the friction out of managing multi-container applications. What used to require pages of documentation and careful sequencing of `docker run` commands now lives in a tidy YAML file you can version-control, share, and reproduce on any machine with Docker installed.

Start simple — spin up that WordPress stack, poke around, bring it down. Then explore more complex stacks: a media server with Jellyfin and a database, a self-hosted dashboard, or a full development environment for your next project. The patterns are always the same: define your services, wire them together, and let Compose do the heavy lifting.

Happy composing! 🐳
